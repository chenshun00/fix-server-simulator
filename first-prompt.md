你是一名熟悉 FIX 协议和交易系统的技术产品经理，
长期与后端工程师、测试、交易系统协作。

现在请你 **输出一份工程导向的 PRD**，
用于指导开发一个「FIX Server Simulator（带 Web 控制台）」。

====================
一、项目背景
====================

- 系统名称：FIX Server Simulator
- 目标用户：后端开发 / 测试 / 联调人员
- 使用场景：
  - 本地开发调试 FIX Gateway
  - 模拟对手方行为
  - 回放与分析 FIX 报文

====================
二、产品目标
====================

【核心目标】
- 稳定接收并记录所有 FIX 报文
- 支持人工与规则化的模拟响应
- 提供清晰、可追溯的 Web 操作界面

【非目标】
- 不实现真实撮合
- 不承担真实生产交易

====================
三、用户与使用流程
====================

### 3.1 使用角色
- 开发人员
- 测试人员

### 3.2 核心使用流程（端到端）
1. Gateway 发起 FIX Session
2. Simulator 建立 Session 并接收报文
3. 报文落库
4. Web 页面查看报文
5. 人工或规则触发回报

====================
四、功能需求（核心）
====================

### 4.1 FIX Session 管理
- 建立 / 断开 Session
- SeqNum 管理
- Heartbeat / TestRequest

### 4.2 报文接收与存储
- 原始 FIX 报文存储
- 解析后字段存储
- 按 Session / 时间 / MsgType 查询

### 4.3 Web 控制台
- Session 列表页
- 报文列表页
- 报文详情页（原文 + 解析字段）
- 人工触发响应（ExecutionReport / Reject）

### 4.4 模拟行为控制
- 成功回报
- 拒单回报
- 延迟回报
- 不回报（模拟异常）

====================
五、非功能性需求
====================

- 可观测性：日志清晰
- 可追溯性：操作可审计
- 可扩展性：未来支持更多 MsgType
- 可测试性：支持自动化测试

====================
六、约束与假设
====================

- FIX 版本：4.4
- 并发规模：单 Gateway / 少量 Session
- 数据库：关系型数据库
- 不考虑高可用

====================
七、验收标准
====================

- Gateway 能成功建立 Session
- 所有收到的报文均可在 Web 页面查看
- 人工触发的 FIX 回报能被 Gateway 接收
